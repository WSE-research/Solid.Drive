= RequestsPanel

Collapsible panel for managing incoming access requests in the
classic layout. Reads its data from
link:../../contexts/RequestNotificationsContext/[RequestNotificationsContext] —
the single source of truth fed by the Solid Notifications WebSocket
subscription — so the panel stays live without owning its own fetch
hook.

== Features

Auto-open on notification:: When the bell calls `selectRequest(uri)` the context bumps `navigationCount`, which the panel watches: the panel opens itself, scrolls into view, and the selected row is highlighted.
Mark seen:: Every request rendered while the panel is open is marked seen via the context, so the bell badge clears.
Approve (catalog):: Creates a per-viewer shared catalog, grants read access to it. Deletes the inbox message on success.
Approve (file):: Grants the requester read access to the specific file container. Deletes the inbox message on success.
Approve (type):: Reads the catalog, walks every entry whose `conformsTo` matches the requested class, and grants read on each container. Tolerates per-entry failures; only fails when every matching container is unreachable.
Discovery access:: Every approve path also calls `ensureDiscoveryAccess` so the requester gains read on the main catalog and list-only on the app container.
Deny:: Sends a rejection notification to the requester's inbox, then deletes the message.
Duplicate sweep:: Approve/deny remove every duplicate inbox message for the same `(requesterWebId, requestType, accessTo, forClass)` tuple — the user only sees and acts on one row even when the sender retried.
Badge:: Shows the count of pending requests on the toggle button.
Requester profile:: Each row shows the requester's avatar and display name via `RequesterRow`.
