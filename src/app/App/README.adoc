= App

Root component. Sets up providers and picks the shell the user sees.

The provider stack is always the same: `<BrowserRouter>` > `<BrowserSolidLdoProvider>` > `<NotificationProvider>` > `<RequestNotificationsGate>`.
What changes is what gets rendered inside it, based on auth state and the saved layout preference.

== Which shell renders

[cols="1,2", options="header"]
|===
| Condition | Renders

| Coming back from the Solid issuer (URL has `code` + `state`) | `<AuthCallbackSkeleton />`
| Logged out | `<LandingPage />`
| Logged in, OneDrive preference | `<OneDriveLayout />`
| Logged in, Classic preference | `<Header />` + `<ClassicLayout />`
|===

A few details worth knowing:

* Session continuity: after a refresh, the previously logged-in shell stays mounted while auth restores. We never flash back to the LandingPage in that case.
* The auth-callback skeleton also covers a short boot window after the handshake resolves, so the next layout has time to start pod discovery before it takes over.
* See link:../hooks/useSessionContinuity/[useSessionContinuity] for the exact logic.
