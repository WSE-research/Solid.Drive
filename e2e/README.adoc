= e2e tests
:toc: macro
:toclevels: 2

End-to-end tests for two flows: category-level access sharing, and the
OneDrive-inspired layout shell. They run real browser contexts (Peach
as owner, Parni as requester) against a local Community Solid Server
and the Vite dev build.

toc::[]

== Quick start

[source,bash]
----
npm run test:e2e         # headless
npm run test:e2e:ui      # Playwright UI mode, best for debugging

# focus on a single file
npx playwright test e2e/tests/category-share.spec.ts

# focus on tests matching a name
npx playwright test -g "Peach revokes"
----

First time only:

[source,bash]
----
npx playwright install chromium
----

On failure, screenshots, video, and trace land under `test-results/`.

Step-by-step screenshots that document each workflow land under
`e2e/screenshots/<test-name>/`. They are off by default locally; run
`E2E_SCREENSHOTS=1 npm run test:e2e` to produce them, or see
`e2e/screenshots/README.adoc` for the full story (CI always captures
them and commits them back on `main`).

== What runs

`playwright.config.ts` boots Vite on `:5173` and Community Solid
Server on `:3001`. CSS is seeded from `e2e/fixtures/seed.json` with
two accounts: `peach@e2e.test` / `peachy` at `/peach/`, and
`parni@e2e.test` / `parnitest` at `/parni/`. Locally,
`reuseExistingServer` is on; CI gets a fresh CSS per job.

== Layout

[cols="1,3", options="header"]
|===
| Path | Purpose

| `playwright.config.ts`
| Test runner config and `webServer` boot.

| `e2e/config.ts`
| Shared knobs for the suite: CSS and app URLs, timeout budgets,
the localStorage key and event names the helpers reach into.

| `e2e/fixtures/seed.json`
| CSS seed config.

| `e2e/fixtures/assets/*.png`
| Real PNGs uploaded as the binary content of seeded files.

2+| **Helpers**

| `e2e/helpers/fixtures.ts`
| Playwright fixtures (`peach`, `parni`) and `freshLogin`. Use this
in new tests. `freshLogin` takes an optional `permissions` array
granted on the browser context.

| `e2e/helpers/css-auth.ts`
| `getAuthenticatedFetch`: CSS account API to client credentials to a
DPoP-bound `fetch`.

| `e2e/helpers/login-ui.ts`
| Drives the Solid OIDC dance through the app's header.

| `e2e/helpers/seed.ts`
| Pod-shaping helpers: profile basics, file upload, ACL grants,
contacts, cleanup. `ensureProfileBasics` patches the triples the
app's profile-driven UI needs (`pim:storage`, `foaf:name`,
`ldp:inbox`, `dcat:catalog`, plus public `acl:Append` on the inbox).
`removeProfileStorage` strips `pim:storage` back out to exercise the
storage-discovery walk.

| `e2e/helpers/ui-actions.ts`
| UI helpers: add contact, request access, approve or deny, revoke
through the share panel, expand a type folder.

| `e2e/helpers/onedrive.ts`
| OneDrive-layout helpers: switch the layout preference, navigate
between rail views, jump to My Files and wait for the table.

| `e2e/helpers/screenshots.ts`
| Step-by-step screenshot capture (`shot`, `step`) and the per-test
folder setup behind the `documentationScreenshots` auto fixture. The
UI helpers above call `shot()` after each action, so tests are
documented with no per-test wiring. See `e2e/screenshots/`.

2+| **Tests, landing and auth bootstrap**

| `e2e/tests/landing-page.spec.ts`
| The logged-out LandingPage: brand, provider picker (incl. HTWK Leipzig
as the default-selected provider), layout picker, custom-issuer
validation, the No-Pod and Video sub-routes, and that clicking Login
actually redirects to the chosen issuer.

| `e2e/tests/pod-discovery.spec.ts`
| Storage discovery fallback: with `pim:storage` stripped from the
profile (the real CSS case), the app walks `Link: rel="type"` headers up
to the pod root and still resolves the user's storage, not the server
root. Uses `removeProfileStorage` to force the fallback.

| `e2e/tests/auth-callback-skeleton.spec.ts`
| The skeleton shown while we are coming back from the issuer: it
appears whenever the URL has `?code&state`, reuses the landing brand
wordmark and HeroBlob, and lifts on its own after the boot window so
the next shell can take over.

2+| **Tests, category-level access**

| `e2e/tests/category-share.spec.ts`
| Type-level request, approve, file appears as shared.

| `e2e/tests/revoke-keeps-visible.spec.ts`

2+| **Tests, OneDrive layout**

| `e2e/tests/onedrive-layout.spec.ts`
| Opting into the OneDrive shell and switching back.

| `e2e/tests/onedrive-navigation.spec.ts`
| The nav rail switches between all five views.

| `e2e/tests/onedrive-topbar.spec.ts`
| Avatar menu, settings menu language switch, log out.

| `e2e/tests/onedrive-my-files.spec.ts`
| Pod browser renders, browsing a folder shows a seeded file,

| `e2e/tests/onedrive-selection.spec.ts`
| Selection action strip, copy link with toast and clipboard, delete

| `e2e/tests/notification-bell.spec.ts`

== Adding a test

Start from the fixture. It authenticates both users and wipes
Peach's pod before every test. Pull `peach` and `parni` and call the
seeders your scenario needs.

[source,typescript]
----
import { test, expect, freshLogin } from "../helpers/fixtures";
import { addContact, grantDiscoveryAccess, seedFile } from "../helpers/seed";
import { TEST_TIMEOUTS } from "../config";

test("describe what is being asserted", async ({ browser, peach, parni }) => {
  test.setTimeout(TEST_TIMEOUTS.medium);

  await seedFile({
    authedFetch: peach.authedFetch,
    pod: peach.pod,
    fileName: `something-${Date.now()}.png`,
    classUri: "http://schema.org/ImageObject",
    mediaType: "image/png",
    title: "My File",
    asset: "Holiday_Photo.png",
  });
  await grantDiscoveryAccess(peach.authedFetch, peach.pod, parni.pod.webId);
  await addContact(parni.authedFetch, parni.pod.webId, peach.pod.webId);

  const session = await freshLogin(browser, parni);
  await session.page.getByRole("button", { name: /Request all Image/i }).click();
  // ...
  await session.close();
});
----

Use the direct fetch (`peach.authedFetch`, `parni.authedFetch`) for
setup: catalog rows, ACLs, contacts, profile triples. Anything that
is not the surface under test goes through fetch. Use the UI for the
assertion: the actual button, the expected text, the disappearance
of a folder.

After the other side changes server state, close the old context and
call `freshLogin` again. The existing context still holds pre-change
auth tokens and LDO subject caches, so reading the new state through
it is unreliable.

`pod` is the source of truth for URIs (`pod.webId`, `pod.appContainer`,
`pod.catalogUri`, ...). Never hard-code `http://localhost:3001/...`
in a test.

== Debugging

UI mode (`npm run test:e2e:ui`) is best for understanding a flake or
building a new test. Step through actions, inspect the network panel
per step, re-run a single test without restarting CSS.

On a headless failure, look under `test-results/` for the failing
test's directory: `test-failed-*.png` screenshots, `video.webm`, and
`trace.zip`. Open the trace with
`npx playwright show-trace test-results/<dir>/trace.zip` for a
timeline of every action, request, and DOM snapshot.

=== Expected noise in the network panel

UI mode shows red entries even on a green run. Most are intentional.

[cols="1,3", options="header"]
|===
| Error | What it means

| `catalog.ttl 404`
| The contact's main catalog does not exist yet, or not for this
contact. `useSharedCatalog` falls back gracefully.

| `.shared-…/profile/card.ttl 403` (twice)
| `useSharedCatalog` probes two candidate URIs with and without the
`#me` fragment. When neither exists, the hook falls through to the
main catalog.

| `index.ttl 403` on a per-file metadata document
| A HEAD on a file the viewer cannot read. That is how the hook
decides whether the file is shared or browsable.

| `.acl 403`
| `loadAcl` fired on a file the viewer does not own. Only happens
when the share panel opens on a non-owned file.
|===

A 404 on the binary URL of a shared file is not on this list, and is
a real bug.

== Not covered, and where it lives instead

* File upload UI (drag and drop, file picker, SHACL validation
feedback): `useFileUpload-test`.
* Contact validation in the Add input: `useContacts-test`.
* Catalog editing and deletion through the UI: `FileCard-test`.
* My Files sort ordering and the full upload form: `MyFilesView-test`,
`MyFilesTable-test`, `sortEntries-test`.
* Concurrent requests from multiple contacts: single-user scenarios
only.
* Federation across multiple Solid pod providers: local CSS only.
* Anything that depends on a full page reload while logged in. The
suite uses fresh browser contexts and never reloads. The session
continuity and URL-read paths have unit coverage instead.
* The Recent, Shared, Requests, and People views are still
view-stub placeholders; the navigation spec asserts page title and
rail switch only.

== Performance

Roughly 3 minutes cold (CSS plus Vite booting fresh) and 1 minute 40
seconds warm. Per spec, 7 to 25 seconds. If a spec drifts much
beyond that, switch to UI mode and look for what is timing out.

Every test logs in through the OIDC redirect dance, and `freshLogin`
runs it again after the other party changes server state. Cutting to
a single client-credentials token or to `storageState` reuse would
shave most per-test overhead, but it has not been worth the work so
far.
