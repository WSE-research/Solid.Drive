= Solid.Drive GitHub Pages site

A small, password-protected marketing page for Solid.Drive. The visual
design follows the
https://github.com/WSE-research/Solid.Sync[Solid.Sync] GitHub page: DM
Sans, a soft lavender gradient background, white cards with gradient icon
chips and soft shadows, and a fixed, semi-transparent header. The
gradient ".DRIVE" wordmark and the animated blue "wave" (fixed to the
right-hand edge of the screen) tie it back to Solid.Drive (issue #69).

It presents a short description of Solid.Drive and its digital-sovereignty
intention (framed around Tim Berners-Lee's Solid vision) with the logo
beside the headline, the provided features, an interactive theme ×
color-scheme screenshot switcher, a gallery of further views (click any
screenshot for a lightbox with previous / next navigation and the image
title shown at the bottom), and links to the live demo, the GitHub
repository, and the companion
https://github.com/WSE-research/Solid.Sync[Solid.Sync].

== Contents

[cols="1,3", options="header"]
|===
| Path | Purpose

| `index.html`
| The page itself. Carries empty Jekyll front matter so it is processed
by the build; all markup is inline, plus a small script for the showcase
switcher and the scroll-to-top button.

| `assets/css/style.css`
| Hand-written theme mirroring the live app's tokens (App.css /
LandingPage.css): light surface, DM Sans, the hero blob, the switcher.

| `assets/img/logo.png`
| The Solid.Drive brand mark, used as the favicon and header logo.

| `assets/img/showcase/`
| `my-files-<theme>-<scheme>.png` — the My Files view in all eight
dark/light × indigo/emerald/amber/rose combinations, generated by
`e2e/tests/theme-color-gallery.spec.ts` and driven by the "See it in
action" switcher.

| `assets/img/gallery/`
| Further view screenshots (copied from `docs/images/`).

| `_config.yml`
| Minimal Jekyll config (no theme, no plugins).

| `_protected_pages.txt`
| PageCrypt manifest: `<page> <password>`, one per line. Read by the
deploy workflow. The password is stored in plaintext here, so it
gate-keeps the page rather than being a strong secret.
|===

== Deployment

Built and deployed by `.github/workflows/pages.yml` on every push to
`main` that touches `gh-page/**`:

. `actions/jekyll-build-pages` builds `gh-page/` into `_site/`.
. PageCrypt (https://github.com/Greenheart/pagecrypt[pagecrypt]) encrypts
  each page listed in `_protected_pages.txt` in place — client-side AES,
  so visitors must enter the password to view the content. This follows
  https://github.com/evanbaldonado/github-pages-jekyll-password-protection[evanbaldonado/github-pages-jekyll-password-protection].
. `actions/deploy-pages` publishes `_site/` to GitHub Pages.

[NOTE]
====
*One-time setup by a repo admin:* enable GitHub Pages with
Settings → Pages → Build and deployment → Source → *GitHub Actions*.
Until then the workflow builds but has nowhere to deploy.

PageCrypt only encrypts the HTML, not its assets (CSS, images), matching
the upstream tool's behaviour — fine here, since the page content is
public-facing marketing.
====

== Local preview

[source,bash]
----
# Emulate the Jekyll build (strip the empty front matter) and serve it:
mkdir -p _preview && cp -r gh-page/assets _preview/
tail -n +3 gh-page/index.html > _preview/index.html
python3 -m http.server -d _preview 8099   # then open http://localhost:8099/
----

Validated by `tests/integration/gh-page.test.ts` (`npm test`): required
content and links, that every referenced asset exists, and that the
PageCrypt + Pages deploy wiring is in place.
